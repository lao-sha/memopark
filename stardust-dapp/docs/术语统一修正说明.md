## 📋 修正背景

### 问题发现

在代码审查和用户测试过程中，发现了以下术语不统一的问题：

| 原术语 | 正确术语 | 问题 | 影响 |
|-------|---------|------|------|
| 逝者姓名 | 纪念馆名称 | 概念混淆 | ⭐⭐⭐⭐⭐ |
| 墓位ID | 纪念馆ID | 术语不统一 | ⭐⭐⭐⭐⭐ |
| 墓地 | 纪念馆 | 传统vs现代 | ⭐⭐⭐⭐ |

### 修正必要性

**1. 概念统一性**
- ✅ 项目名称：Stardust（星尘纪念）
- ✅ 核心概念：数字纪念馆（Digital Memorial Hall）
- ✅ 技术特点：去中心化、区块链、IPFS
- ❌ 传统术语："墓位"、"墓地"带有物理空间局限

**2. 用户认知**
- 传统："墓位"让人联想到墓园、墓碑
- 现代："纪念馆"符合数字化纪念理念
- 提升：更符合虚拟世界的永恒纪念概念

**3. 业务逻辑**
- Grave = 纪念馆（Memorial Hall）
- Deceased = 逝者（Deceased Person）
- 关系：一个纪念馆可包含多个逝者

### 正式术语表

| 中文术语 | 英文术语 | 说明 | 使用场景 |
|---------|---------|------|----------|

| ❌ 墓地 | ✅ 纪念馆 | 物理vs虚拟 |
| ❌ 墓主 | ✅ 管理员/馆主 | 概念不清晰 |

### 术语使用示例

#### ✅ 正确用法

**页面标题：**
- "创建纪念馆"（而非"创建墓地"）

**表单字段：**
- "纪念馆名称"（而非"墓位名称"）

- "关联纪念馆"（而非"关联墓位"）

- "选择纪念馆"（而非"选择墓位"）
- "纪念馆列表"（而非"墓位列表"）

## 📖 概念对照表

### Stardust 核心概念

| 中文 | 英文 | Pallet | 说明 |
|-----|------|--------|------|
| 纪念馆 | Grave / Memorial Hall | memo_grave | 纪念空间/容器 |
| 逝者 | Deceased | deceased | 具体人物档案 |
| 园区 | Park | memo_grave | 纪念馆分组 |
| 供奉 | Offering | offerings | 虚拟祭品 |
| 关系 | Relationship | deceased | 逝者间的亲属关系 |

### 详细说明

**纪念馆（Grave）：**
- 定义：数字纪念空间的容器
- 类比：现实中的家族祠堂、纪念堂
- 特点：可包含多个逝者档案
- 功能：组织和展示纪念内容

**逝者（Deceased）：**
- 定义：具体的逝者个人档案
- 类比：现实中的墓碑信息
- 特点：详细的个人信息
- 功能：存储和展示逝者资料

**园区（Park）：**
- 定义：纪念馆的分组
- 类比：现实中的墓园、陵园
- 特点：可包含多个纪念馆
- 功能：统一管理和展示

## 📝 命名规范

### 纪念馆命名

**个人纪念馆：**
```
格式：{姓名} + 纪念馆/纪念堂/纪念园

示例：
✅ 张三纪念馆
✅ 李四先生纪念堂
✅ 王五女士纪念园
✅ 赵六老先生纪念馆
```

**家族纪念馆：**
```
格式：{姓氏} + 祠堂/宗祠/纪念园

示例：
✅ 李氏祠堂
✅ 陈氏宗祠
✅ 王家纪念园
✅ 赵氏家族馆
```

**夫妻纪念馆：**
```
格式：{夫姓名}{妻姓名} + 纪念馆

示例：
✅ 张三李四纪念馆
✅ 王五赵六合葬墓
✅ 陈某某某纪念堂
```

### 逝者命名

**基本原则：**
```
使用逝者的真实姓名

示例：
✅ 张三
✅ 李四先生
✅ 王五女士
✅ 赵六老先生

避免：
❌ 昵称（小张）
❌ 网名
❌ 代号
```
